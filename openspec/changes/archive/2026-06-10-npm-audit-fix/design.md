# Design: npm-audit-fix

## Technical Approach

Zero-code-change configuration: add an `overrides` block to `package.json` that forces two transitive dependencies (`serialize-javascript`, `cookie`) to their patched versions. npm overrides tell the resolver "use this version regardless of what the parent declares in its semver range." After the override, run `npm install` to regenerate `package-lock.json`, then verify with `npm audit` and the full test suite.

## Architecture Decisions

### Decision: npm `overrides` vs `resolutions` vs `npm audit fix --force`

| Option | Tradeoff | Decision |
|--------|----------|----------|
| `overrides` (npm) | Standard npm mechanism, no extra tooling | **Selected** — native to npm, no deps |
| `resolutions` (Yarn) | Same concept, different field name | Rejected — project uses npm, not Yarn |
| `npm audit fix --force` | Auto-upgrades direct deps to breaking majors | Rejected — would bump mocha 12.x and @sveltejs/kit past v2 |

**Rationale**: `overrides` is the npm-native way to pin transitive deps. It's a single JSON key with no tooling, no plugins, and no side effects beyond the dependency tree.

### Decision: Pin to exact versions (no ranges)

| Option | Tradeoff | Decision |
|--------|----------|----------|
| `"7.0.5"` | Exact pin, no surprise upgrades | **Selected** — security fix should be deterministic |
| `"^7.0.5"` | Allows minor/patch bumps | Rejected — defeats the purpose of a targeted fix |

**Rationale**: The audit reported specific CVEs fixed in `serialize-javascript@7.0.5` and `cookie@0.7.2`. Using range syntax would negate the guarantee. If newer patches are needed, a future change updates the override.

## Data Flow

```
package.json (overrides)
        │
        ▼
npm install ──→ package-lock.json (serialize-javascript@7.0.5, cookie@0.7.2)
        │
        ├──→ npm audit (exit 0, 0 vulnerabilities)
        └──→ npm test (all pass)
```

No runtime data flow changes — this is a build-time dependency resolution fix.

## File Changes

| File | Action | Description |
|------|--------|-------------|
| `package.json` | Modify | Add `overrides` section with two entries |
| `package-lock.json` | Modify | Regenerated by `npm install` — lock commits with the change |

## Interfaces / Contracts

No new interfaces. The `overrides` key in `package.json` follows the [npm overrides spec](https://docs.npmjs.com/cli/v11/configuring-npm/package-json#overrides):

```json
{
  "overrides": {
    "serialize-javascript": "7.0.5",
    "cookie": "0.7.2"
  }
}
```

## Testing Strategy

| Layer | What to Test | Approach |
|-------|-------------|----------|
| Dependency | Override resolution | `npm ls serialize-javascript` shows `7.0.5`; `npm ls cookie` shows `0.7.2` |
| Security | Zero vulnerabilities | `npm audit` must exit 0 with "0 vulnerabilities" |
| Unit | Vitest suite | `npm test` passes |
| Lint | Svelte check | `npm run check` exits 0 |
| E2E | WDIO + mocha | `npm run test:e2e` passes (verifies mocha still works with serialize-javascript@7) |

## Migration / Rollout

No migration required. Rollback: revert the commit adding overrides, run `npm install` to restore original lockfile.
